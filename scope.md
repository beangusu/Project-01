# Project Scope Statement

**Project Name:** Personal Introduction Webpage

## Project Purpose

> *Why are we doing this project?*

We are doing this project to apply the project management principle of scoping to a real, hands-on technical task! By defining purpose, goals, and limitations before writing a single line of HTML, we are practicing the *planning discipline* that makes project execution more focused and efficient. The project will also serve as a foundational exercise in the course, bridging theoretical project management concepts from *Project Managing for the Unofficial Project Manager* with practical web development skills.

## Description

> *How would you describe this project?*

This project involves building a single-page personal introduction website using semantic HTML!

The page will:

- Include a `<title>` element in the document `<head>`
- Display my name in an `<h1>` heading and my major *or* professional interests in an `<h2>` subheading
- Contain *at least* two `<p>` paragraphs describing my goals for this course and my career
- Feature an unordered list (`<ul>`) with at least *three* `<li>` items covering my hobbies or interests
- Link to my GitHub profile using an `<a>` tag

The project also includes this `scope.md` file, written in Markdown, following the Project Scope Statement template from PMUPM Chapter 3. Both files will be submitted together as a single `.zip` archive.

## Desired Results

> *What specific outcomes must we achieve to be successful?*

- A valid `index.html` file that satisfies all structural requirements: `title`, `h1`, `h2`, at least two `p` elements, a `ul` with at least three `li` items, and an `a` hyperlink to my GitHub profile
- A complete `scope.md` file written in Markdown that follows the PMUPM Chapter 3 Project Scope Statement template, with thoughtful entries in the "Project Purpose", "Description", "Desired Results", and "Exclusions" sections
- Both files packaged into a single `.zip` and submitted on time, meeting every minimum expectation listed in the assignment guidelines

## Exclusions

> *What is out of the project's scope and should **NOT** be included?*

- Any CSS styling or visual design
- JavaScript or any interactive functionality
- Multiple pages or external assets such as images or third-party frameworks
- Backend components, databases, or any form of hosting or deployment
- Any content or features beyond what the assignment rubric explicitly requires

## Feedback Loops

> *Who needs to provide feedback? When and how?*

- **Instructor** - Reviews and grades both submitted files after the deadline; feedback delivered through the course grade book
- **Self-Review** - Before submission, validate `index.html` by checking it line-by-line against the assignment checklist to confirm all required elements are present and correctly structured

## People Resources

> *Who are the key team members?*

- **Student** - Sole contributor; responsible for writing, reviewing, and submitting both files
- **Instructor** - Evaluator; provides grading and feedback after submission

## Priorities

> *How would you prioritize these constraints? (1 = most important)*

| Constraint | Priority |
|------------|----------|
| Time       | 1        |
| Quality    | 2        |
| Budget     | N/A      |

- **Time:** Submitting before the deadline is the top priority as a late submission risks a zero regardless of quality
- **Quality:** Both files must meet all minimum expectations in the rubric; the HTML must be valid and the scope statement must be complete and follow the Chapter 3 template
- **Budget:** Not applicable — this project has no financial cost

## Acceptance Criteria

> *Who needs to sign off on the project — and when?*

The instructor signs off at grading. The project is considered "done" when both `scope.md` and `index.html` are zipped and submitted, and every item on the assignment checklist is satisfied; including all required HTML elements, the complete scope statement sections, and on-time delivery.
